# LoCoMo Dual-Mode Evaluation Design

## Goal

Improve the LoCoMo benchmark flow so Muninn can be evaluated through the same core pipeline in two modes:

- `diagnostic`: small-sample, high-detail runs for debugging recall, answer context, LLM answers, and scoring.
- `benchmark`: larger runs for version-to-version comparison with aggregate metrics and bounded diagnostics.

Both modes must use the same evaluation path:

`ingest -> observe -> recall -> build answer context -> LLM answer -> score -> report`

This avoids the current split where recall is measured but final answers are generated by a heuristic extractor that does not represent the intended LLM + recall product behavior.

## Non-Goals

- Do not optimize specifically for LoCoMo facts in the generic observer prompt.
- Do not introduce a separate benchmark-only memory layer.
- Do not replace Muninn recall with direct LoCoMo dataset lookup.
- Do not make `--expand-references` the default.
- Do not require a full LoCoMo run before the diagnostic mode is useful.
- Do not remove the existing heuristic answerer immediately; keep it as a baseline field during the transition.

## Current Problems

The current LoCoMo runner imports turns, waits for observing, performs batch recall, and then builds answers with heuristic string extraction.

This has three issues:

1. Recall and answering are conflated.
   A question can have perfect evidence recall but still get a bad answer because the heuristic extractor reads wide snapshot details or dates incorrectly.

2. The matched semantic text is discarded.
   `coreClient.memories.recall()` returns the semantic row text that actually matched the query, but the bridge currently resolves only the rendered memory snapshot. The Python side receives broad `title`, `summary`, and `detail` diagnostics, not the exact matched memory text.

3. Diagnostic output is insufficient.
   The result files show prediction, evidence ids, and hit ids, but they do not fully show the query candidates, matched memory text, answer context given to the answerer, final answer, and per-question scoring in one place.

## Modes

### Diagnostic Mode

Diagnostic mode is the default development mode.

It should optimize for explainability:

- Small sample count.
- Optional `--limit-questions`.
- Full per-QA trace output.
- Full recall hit diagnostics.
- Full answer context shown to the LLM answerer.
- Kept home support through `--keep-home`.
- Clear progress events for import, recall, answer, score, and report phases.

Expected command shape:

```bash
python3 benchmark/locomo/run.py \
  --data-file benchmark/locomo/out/conv-26-session-1.slice.json \
  --out-file benchmark/locomo/out/conv-26.diagnostic.json \
  --mode diagnostic \
  --limit-questions 4 \
  --top-k 5 \
  --keep-home
```

### Benchmark Mode

Benchmark mode is for larger comparison runs.

It should optimize for stable aggregate outputs:

- Same pipeline as diagnostic mode.
- Aggregate stats remain first-class.
- Trace output is still produced but can be smaller or sampled if needed later.
- Progress output remains machine-readable.
- Mode metadata records the run shape so results can be compared across versions.

Benchmark mode must not use a different answer-building path from diagnostic mode.

## Pipeline

### 1. Ingest

Import LoCoMo dialog turns through the existing Muninn bridge.

The manifest should continue to record:

- `turn_id`
- `source_id`
- `sample_id`
- `session_id`
- `date_time`
- `import_order`

The manifest is diagnostic/scoring metadata. It must not silently replace the model-visible answer context.

Date injection into observer-visible turn content is a related benchmark improvement, but it should be implemented as an explicit ingest behavior rather than hidden scoring logic.

### 2. Observe

The runner continues to wait for observer watermark resolution before recall.

Timeout diagnostics should keep showing:

- target turn id
- pending turn ids
- observing epoch
- committed epoch

### 3. Recall

The bridge should preserve the exact semantic match from `coreClient.memories.recall()`:

- `matched_text`: the semantic index row text selected by recall
- `memory_id`: the rendered memory id
- `evidence_ids`: resolved LoCoMo evidence ids
- `date_time`: dates resolved from direct evidence references through the manifest
- `title`, `summary`, `detail`: rendered memory snapshot diagnostics

The Python `RecallHit` type should expose `matched_text`.

Deduplication should still happen by `memory_id` when merging query-candidate recall results.

### 4. Build Answer Context

Each QA should get an explicit answer context built from recall hits.

Default context, with `--expand-references` disabled:

- question
- category
- top-k hit list
- for each hit:
  - `DATE`
  - `MEMORY_ID`
  - `MATCHED_MEMORY`
  - `EVIDENCE_IDS`

Expanded context, with `--expand-references` enabled:

- includes the default context
- expands only direct `session:*` turn references
- formats expanded references as `DATE` plus conversation turn text
- does not recursively expand observing references

This keeps default runs focused on what Muninn actually recalled while still allowing direct evidence inspection when needed.

### 5. LLM Answerer

The LLM answerer becomes the primary prediction path.

It should:

- Reuse the LLM configuration from `muninn.json`.
- Run once per QA.
- Receive only the structured answer context for that QA.
- Return a short JSON object containing the answer.
- Avoid explanations unless a future flag explicitly asks for rationales.

The answer prompt should instruct:

- Answer only from the provided memories/context.
- If the answer is not present, return `Not mentioned in the conversation`.
- For date questions, use provided `DATE` values as anchors for relative time when the memory text makes that relationship clear.
- Keep the answer short because existing LoCoMo scoring expects concise answers.

### 6. Heuristic Baseline

The existing heuristic prediction should remain as a baseline field during this iteration.

It should not be the primary `model_key` used for stats once LLM answerer is enabled.

Proposed output fields:

- `muninn_top_5_prediction`: primary LLM answer
- `muninn_top_5_heuristic_prediction`: previous heuristic answer
- `muninn_top_5_context`: evidence ids used for recall scoring
- `muninn_top_5_hits`: compact hit diagnostics

### 7. Score

Scoring should keep the existing LoCoMo F1 and evidence recall behavior.

The important separation is:

- recall score measures whether gold evidence ids appeared in context ids
- F1 score measures whether the answerer produced the gold answer

The report should explicitly distinguish:

- recall miss: evidence was not retrieved
- answer miss after recall: evidence was retrieved, answer was wrong
- adversarial conflict: answer matched adversarial answer

## Output Files

The runner should write:

- `out.json`: LoCoMo-style sample results, with primary predictions and compatibility fields.
- `_stats.json`: aggregate F1 and recall by category.
- `_report.json`: ranked error analysis.
- `_trace.json`: full diagnostic trace by sample and QA.
- `_metadata.json`: run mode, answerer model/config summary, top-k, expand references setting, timing, and input/output paths.
- optional progress JSONL: existing progress events plus answerer phases.

## Trace Shape

Each QA trace should include:

- `sample_id`
- `qa_index`
- `question`
- `category`
- `gold_answer`
- `evidence`
- `query_candidates`
- `hits`
- `answer_context`
- `prediction`
- `heuristic_prediction`
- `f1`
- `recall`
- `adversarial_answer`
- `adversarial_match`

Each hit in trace should include:

- `memory_id`
- `matched_text`
- `evidence_ids`
- `date_time`
- `title`
- `summary`
- optionally `detail` in diagnostic mode

## CLI

Add:

- `--mode diagnostic|benchmark`, default `diagnostic`.
- `--expand-references`, default false.
- `--answerer llm|heuristic`, default `llm` for the new flow.

Keep:

- `--data-file`
- `--out-file`
- `--progress-file`
- `--top-k`
- `--sample-id`
- `--limit-questions`
- `--keep-home`

If `--answerer llm` is selected and no usable LLM config is available, the runner should fail fast with a clear error.

## Testing Strategy

Use deterministic tests for the flow boundaries:

- Bridge tests verify recall hit conversion preserves `matched_text`.
- Python unit tests verify answer context formatting with and without `--expand-references`.
- Python unit tests verify trace records include question, hits, answer context, predictions, and scores.
- Python unit tests verify `--mode diagnostic` and `--mode benchmark` choose the same pipeline and differ only in output detail defaults.
- Existing scoring tests continue to validate F1 and evidence recall.

Live LLM tests are not required for the default test suite. A later opt-in live benchmark run can validate the real configured answerer.

## Risks

- LLM answers are non-deterministic. The trace must show enough context to debug whether failures are recall failures or answerer failures.
- If `matched_text` is too narrow, answer quality may drop without `--expand-references`. That is useful signal for memory quality, not a reason to default to expansion.
- If `--expand-references` is overused, the benchmark can become a broad evidence reader instead of a memory recall benchmark.
- Reusing `muninn.json` LLM config can accidentally depend on local secrets. Tests should mock answerer behavior and avoid printing secrets.
